# Looper

Autonomous AI software delivery team on AutoGen AgentChat (0.7.x) `GraphFlow`. Nine agents (PM, Scope Validator, Architect, FE/BE/OPS engineers, QA, UAT, Release Reporter) run a deterministic pipeline with QA rework loops (max 3) and a single UAT re-scope loop. The authoritative behavior spec is `SPEC.md` — keep prompts and routing consistent with it. The Release Reporter exists partly for a structural reason: GraphFlow validation requires at least one leaf node, and the run completes naturally when it executes.

## Commands

```sh
.venv/bin/pip install -r requirements.txt          # deps
PYTHONPATH=src .venv/bin/python -m looper.main "<goal>"   # run pipeline
PYTHONPATH=src .venv/bin/python -m looper.main --resume output/checkpoints/<stamp>.json   # resume a failed/interrupted run
PYTHONPATH=src:. .venv/bin/python -m unittest discover -s tests -t . -v   # orchestration tests (no Claude Code sessions, instant)

PYTHONPATH=src:. .venv/bin/uvicorn server.app:app --reload --port 8000   # web UI backend
cd web && npm install && npm run dev                                     # web UI frontend (http://localhost:5173)
```

All nine agents run via `claude-agent-sdk`, which shells out to the `claude` CLI (Claude Code) and uses **its own OAuth login** — no `ANTHROPIC_API_KEY`/`OPENAI_API_KEY` needed. Requires the `claude` CLI installed and authenticated on the host (`claude auth status`). Optional `LOOPER_CODE_MODEL` overrides the model all agents use (default: `"sonnet"` — a `claude` CLI alias for the latest Sonnet model, per `claude --help`; deliberately not Claude Code's own default, which is typically Opus-tier and materially more expensive across the many real sessions one run consumes). `LOOPER_CODE_MODEL_<AGENT_NAME>` (uppercased, e.g. `LOOPER_CODE_MODEL_RELEASE_REPORTER=haiku`) overrides it per role; defaults are deliberately uniform — reasoning turns are only ~7% of a real run's cost, and the roles cheapest to downgrade (PM/architect) are the ones whose quality steers everything downstream.

`src/looper/config.py` (`make_model_client`, `LOOPER_PROVIDER`) is unused dead code, kept intentionally in case the pipeline is ever wired back to a raw-API-key provider (`AnthropicChatCompletionClient`/`OpenAIChatCompletionClient` via `autogen-ext`) instead of the `claude` CLI. It is not called from `pipeline.py` or `main.py` — don't assume it's live.

## Architecture

- `src/looper/pipeline.py` — the graph. Key invariant: nodes with multiple incoming edges (engineers: architect + QA-rework; PM: entry + UAT-reject) must keep both edges in a shared `activation_group` with `activation_condition="any"`, or the first pass deadlocks (GraphFlow defaults to "all").
- Routing is by verdict tokens (`QA_PASS`/`QA_FAIL`, `UAT_APPROVED`/`UAT_REJECTED`) matched against the **last non-empty line** of the agent's message (`verdict_is` / `last_line`) — never substring-match, since prompts/templates can mention both tokens.
- `src/looper/termination.py` — `TokenCountTermination` hard-stops after 3 `QA_FAIL`s with a `PIPELINE_FAILED` stop reason.
- `src/looper/prompts.py` — verbatim from SPEC.md section 3 (including the "token must appear nowhere else" hardening on QA/UAT). Edit SPEC.md and prompts.py together; they must not drift.
- `src/looper/claude_code_agent.py` — `ClaudeCodeAgent`, a `BaseChatAgent` that runs its role as a real `claude_agent_sdk` session (`permission_mode="acceptEdits"`) instead of a plain model-client completion. Every agent is one of these. PM, Architect, UAT, and Release Reporter get `allowed_tools=[]` (pure text reasoning, no filesystem access); FE and BE get file/bash tools scoped to `workspace/frontend/`/`workspace/backend/`; OPS gets the workspace root; QA gets `Read`/`Glob`/`Grep`/`Bash` only (**no** `Write`/`Edit` — it verifies, it doesn't modify the code it's independently checking) scoped to the root. See README's "Access per role" table for the full breakdown. Each `query()` call is a fresh Claude Code session with no memory of prior turns, so `ClaudeCodeAgent` replays the accumulated upstream chat history as the prompt every turn (mirroring `AssistantAgent`'s internal `model_context`) — but code from earlier turns persists for the tool-using agents because it's really on disk in the workspace.
- **`tools=` is what restricts availability — `allowed_tools=` does not, despite the name.** Confirmed live: a session built with `allowed_tools=["Read"]` and no `tools` restriction could still run `Bash`. Per `ClaudeAgentOptions`'s own docstrings, `allowed_tools` only skips the permission prompt for tools already available; `tools` (or `disallowed_tools`) is what actually removes a tool from the model's context. `ClaudeCodeAgent.on_messages()` sets both `tools=self._allowed_tools` and `allowed_tools=self._allowed_tools` for this reason — before this fix, every role (including the four "tool-less" reasoning agents) had Claude Code's full default toolset available the whole time, gated only by whether the prompt happened to ask for it. If you add a new tool restriction anywhere, set `tools=`, not just `allowed_tools=`.
- **QA's "no Write/Edit" is not airtight while it still has Bash.** Verified live: with `Write`/`Edit` removed from `tools`, a QA-like agent correctly refused a `Write` call — but then offered to create the same file via `Bash` instead (e.g. `echo ... > file`), since Bash can replicate file-writing and isn't blocked by the `_BASH_DENY_PATTERNS` denylist (blocking generic redirection would also block QA's legitimate build/test output). QA needs Bash to run real tests, so this residual gap is inherent, not a bug to fix trivially — treat it as a prompting-level expectation (QA's prompt says its job is to verify, not modify), not a hard guarantee.
- **Permission guard** (`_make_hooks` in `claude_code_agent.py`): a `PreToolUse` hook, not `allowed_tools` or `can_use_tool`, enforces per-call restrictions (Write/Edit path containment to the agent's own `cwd`; a Bash command denylist). Load-bearing detail if you touch this: `can_use_tool` was tried first and doesn't work — an `allowed_tools` entry that grants a whole tool (e.g. plain `"Bash"`) auto-approves every call before `can_use_tool` is ever consulted (`CanUseToolShadowedWarning`), confirmed live when an unrestricted `curl` call sailed straight through it. `PreToolUse` hooks aren't shadowed this way, which is the entire reason the guard is built on hooks instead. If you add a new restriction, add it as a hook, not a `can_use_tool` callback.
- **Context routing** (`context_sources` on `ClaudeCodeAgent`, configured per role in `pipeline.py`): GraphFlow broadcasts every message to every agent, so an unfiltered history replay grows with the whole transcript and re-sends artifacts a role's prompt says it doesn't consume (plus superseded PRDs/QA reports on loops — a token *and* stale-context problem). Each role's replayed prompt is filtered to the sources its SPEC.md "Input:" line names, keeping only the latest message per source. Filtering happens at prompt-build time (`_build_prompt()`) — `_history` and checkpoints stay complete. Keep the `context_sources` dicts in sync with each role's "Input:" line if either changes.
- **Pointer routing** (`pointer_files` on `ClaudeCodeAgent`; tool-using roles only): `main.py` persists each agent's latest message to `<workspace>/.pipeline-docs/<source>.md` (dir name shared via `pipeline.ARTIFACT_DIR_NAME` — keep `main.py` and `pipeline.py` agreeing on it). On a turn where a pointer-listed artifact did **not** arrive in that turn's delta and its file exists, `_build_prompt()` replays a one-line pointer to the file instead of the full text; the agent Reads/Greps back only what it needs. The effectiveness guardrail is load-bearing: an artifact is always inlined in full the first time it reaches an agent (first pass, or a redesign after UAT re-scope — it's in the delta both times), so pointers can never hide a contract the agent hasn't seen; missing file falls back to inline; `_build_prompt()` with no delta info never pointers. Never set `pointer_files` on a tool-less role — no Read tool means a pointer is a dead end. Measured on a second-QA-round engineer turn: unfiltered replay 26.3K chars → context routing 11.1K (−58%) → routing + design pointer 3.3K (−87%); realized savings are somewhat less since the agent reads back the sections it needs, which is the point — spend scales with what the defects touch, not with design size.
- Workspace layout, built per run in `main.py` under `<output_dir>/workspace/<timestamp>/` and passed into `build_team()`: `frontend/` (FE's cwd), `backend/` (BE's cwd), and the root (OPS's and QA's cwd, so OPS can reference `./frontend` and `./backend` and QA can read/test everything; also the cwd for the tool-less reasoning agents, who never use it). This convention is encoded into the FE/BE/OPS/QA prompts in `prompts.py`/`SPEC.md` — the path names and pipeline.py's directory assignment must stay in sync.
- **Checkpoint/resume** (`main.py`): after every completed agent turn, `main.py` calls `GraphFlow.save_state()` and overwrites `<output_dir>/checkpoints/<stamp>.json` (goal, workspace path, full team state). On any exception (Claude Code session limit hit, crash, etc.) the run stops with that path printed — rerun with `--resume <path>` to continue from the last completed turn instead of restarting at PM. Verified: a freshly-constructed team (new `GraphFlow`, new agent instances) correctly resumes at the next graph node after `load_state()`, without re-running completed ones. This relies on two things staying correct together: `GraphFlow.save_state()`/`load_state()` (built into AutoGen, captures graph position + each participant's pending message buffer) and `ClaudeCodeAgent.save_state()`/`load_state()` (added here — `BaseChatAgent`'s default is a stateless no-op, so without this override the agent's replayed `_history` would not survive a resume even though the graph position would). Resuming calls `run_stream(task=None)` — per AutoGen's own contract, this continues the previous task rather than starting a new one.
- **Resuming a crash that happened mid-fan-out (one of FE/BE/OPS crashes while its siblings succeed) needed a second fix beyond the checkpoint/resume mechanism above** — a real production incident: resuming such a checkpoint completed with exit code 0 but made zero progress (no new messages, no new files), even though the crashed engineer clearly still had pending work. Root cause, found by reading AutoGen's own `GraphFlowManagerState` (`_digraph_group_chat.py`): `select_speaker()` drains its `_ready` queue and resets a dispatched node's activation bookkeeping the INSTANT it's selected to speak — before that node's agent has actually run, let alone completed. There is no "dispatched but not finished" state in GraphFlow's own checkpoint, and the crashed node's parent (e.g. `solution_architect`) already sent its one and only message, so nothing will ever re-enqueue it on its own. Fixed with two pieces: (1) `ClaudeCodeAgent._turn_in_progress` (`claude_code_agent.py`) — True from the moment `on_messages()` accepts a turn until it returns successfully, included in `save_state()`/`load_state()` so it survives the crash; `tests/mock_agent.py`'s `ScriptedClaudeCodeAgent` mirrors the same bookkeeping so mock-based tests exercise it too. (2) `recover_stuck_agents()` (`pipeline.py`) — scans a raw checkpoint dict for any agent whose saved state still has `turn_in_progress: True` and appends it back onto `GraphManager`'s saved `ready` list (a plain list `select_speaker()` just pops from) before `load_state()` runs; `main.py`'s resume path calls it and logs what it recovered. The crashed agent's own `message_buffer` (a separate, container-level list) still holds the exact messages it needs, since `ChatAgentContainer` only clears that buffer on a *successful* completion, never on a crash. **A narrower, separate race exists and is NOT fully fixed**: reproducing this live surfaced that GraphManager's own `message_thread`/`remaining` bookkeeping can itself lag behind messages that already reached the output stream — confirmed by instrumenting `update_message_thread()` directly, a completed sibling's response was visible in the run's output yet never made it into `message_thread` before the crash killed the runtime, because `_log_message` (which feeds the output stream) runs *before* the manager-level `publish_message(GroupChatAgentResponse...)` that actually updates `remaining`/`ready`. This is real but only manifests when a crash happens on the very same runtime tick as a concurrently-completing sibling's response — a window that's microseconds wide, dwarfed by how long a real, slow `claude_agent_sdk` session takes to fail (see `tests/mock_agent.py`'s `_Crash` docstring, which reproduces it with instant scripted responses and works around it in tests with a small `asyncio.sleep()` before raising, to match realistic timing rather than mask a real gap). Treated as an accepted, exceedingly-narrow residual risk rather than something requiring a full reconciliation of GraphManager's derived state against ground truth — re-evaluate if it's ever observed against a real run.
- **`on_event` hook** (`OnEvent` type alias in `claude_code_agent.py`): an optional constructor param on `ClaudeCodeAgent`, fired via `self._emit(event_type, detail, **extra)` at `turn_started`, each `tool_call`, `turn_completed`, and `error`. `_emit` swallows exceptions from the callback itself — a live-status sink must never break an actual pipeline turn. `build_team()` threads `on_event` through to every `agent_cls(...)` call; `main.py` never passes one (`None`, a no-op), so the CLI path is unaffected. This exists for `server/pipeline_runner.py` (below) to record live per-agent progress; `tests/mock_agent.py`'s `ScriptedClaudeCodeAgent` also calls `_emit` at the same points, so the event stream is unit-testable without spending Claude Code quota.
- **`error_max_turns` is not a crash and must not kill the run — but the SDK raises it twice, differently.** Real production incident: two runs of a moderately complex full-stack goal died right after `solution_architect` because one of FE/BE/OPS hit `max_turns` (20) and `on_messages()` treated it identically to a genuine crash. Fix has two parts, both in `claude_code_agent.py`: (1) a `ResultMessage` with `subtype == "error_max_turns"` no longer raises — `transcript` (the `TextBlock`s already seen) becomes `final_text` and the turn completes normally, so QA gets a chance to see and react to the real, incomplete files instead of the whole pipeline dying. (2) **Non-obvious and load-bearing:** confirmed live that after yielding that `ResultMessage`, the CLI subprocess exits non-zero (documented in the SDK's own internal comment: "for shell-script consumers"), which the SDK surfaces as a **second, separate** raw `Exception` on the *next* iteration of the same `async for msg in query(...)` loop — bypassing the `elif isinstance(msg, ResultMessage)` branch entirely, since it's raised at the generator's iterator-protocol level, not inside the loop body. A fix that only branches on `msg.subtype` inside the loop looks correct and passes a shallow read, but still crashes — confirmed by testing against the real SDK with `max_turns=3` before trusting it. The `async for` is now wrapped in `try/except`, gated by a `hit_max_turns` flag set when the first exception (the real ResultMessage) is seen, so the second (redundant) exception is swallowed only in that case — any other exception still raises and fails the run as before.
- **QA/UAT verdict safety net** (`ClaudeCodeAgent._apply_verdict_safety_net()`): `qa_engineer`/`uat_reviewer` MUST end their message in `QA_PASS`/`QA_FAIL`/`UAT_APPROVED`/`UAT_REJECTED` or `verdict_is()` in `pipeline.py` has no edge to route on and that branch of the graph stalls with no error. A turn truncated by `error_max_turns` (above) is the main way this happens, but the check runs unconditionally as a general safety net, not only after a detected max-turns case. Pulled into its own method — pure text logic, no SDK dependency — specifically so it's unit-testable without a Claude Code session; see `tests/test_max_turns_handling.py`. Never silently defaults to PASS/APPROVED: the forced fallback is always the conservative option (`QA_FAIL`/`UAT_REJECTED`).
- **Dynamic per-engineer turn budget** — the root-cause fix for the `error_max_turns` incident above, not just a graceful-degradation patch: a fixed 20-turn budget was too low for the task that crashed, so `ENGINEER_MAX_TURNS` is now only the *starting* value, resized once real complexity is known. `ARCHITECT_PROMPT`'s "Turn Budget Estimate" section (`prompts.py`/`SPEC.md`, kept in sync as always) asks the architect for a numeric FE/BE/OPS estimate; `parse_turn_budget()` and `apply_turn_budget_from_architect()` (`pipeline.py`) extract and apply it via `ClaudeCodeAgent.set_max_turns()` — a new public mutator, since `_max_turns` is otherwise private. Estimates are clamped to `[MIN_ENGINEER_TURNS, MAX_ENGINEER_TURNS]` = `[8, 50]`. Verified live end to end: a real architect session given a URL-shortener PRD (comparable to the goal that originally crashed) produced `FE: 30, BE: 42, OPS: 25`; separately confirmed `set_max_turns()` actually reaches the real SDK session (an agent asked to report its own budget echoed the number set moments earlier, not the constructor default) — `on_messages()` now always states the agent's *current* `self._max_turns` in its system prompt (skipped for tool-less roles, where the cap is a runaway guard, not a real budget) so the model can pace itself. **Load-bearing consequence: `build_team()` now returns `(GraphFlow, dict[str, ClaudeCodeAgent])`, not just `GraphFlow`** — every call site (`main.py`, `server/pipeline_runner.py`, `tests/`) had to be updated to unpack both; a new call site that only unpacks one value will fail loudly (wrong type), not silently. **Resume gotcha, and a real bug in the original fix for it:** `set_max_turns()` overrides are pure in-memory state, not part of `save_state()`/`load_state()` (only `_history` is) — a resumed run's freshly-built engineer agents would silently revert to the static default unless re-derived. The original fix searched an engineer's own replayed `_history` for the architect's message (`find_message_from()`, since removed) — this looked correct and passed all mock-based tests, but broke on a real run: `_history` is only populated when `on_messages()` is actually **called**, and checkpointing (inside `main.py`'s `on_message()`) is synchronous per completed message while the FE/BE/OPS fan-out is dispatched asynchronously. Reproduced live: an architect estimated `FE: 38, BE: 38, OPS: 22`; all three engineers started real sessions (confirmed in `sdk-interactions.jsonl` — `max_turns: 38/38/22` on their first log entries) but all three hit the account-wide Claude Code session limit before any completed, so the checkpoint that ended up on disk was the one saved right after the architect's turn — before any engineer's `on_messages()` had been invoked even once. On `--resume`, GraphFlow correctly re-dispatched all three (its own state had nothing to patch — see `recover_stuck_agents()` above, which found nothing to recover here either, correctly), but `find_message_from()` on any of them returned `None` since none had `_history` yet, so the budget re-application silently no-op'd and every engineer reran at the static `ENGINEER_MAX_TURNS` default (20) — two of them (`frontend_engineer`, `backend_engineer`) then hit *that* ceiling too, on turns already proven to need ~38. Fixed by `reapply_turn_budget_on_resume()` (`pipeline.py`): read the architect's TECH DESIGN back from its on-disk pointer file (`<workspace>/.pipeline-docs/solution_architect.md`, written unconditionally by `on_message()` the instant the architect's turn completes, regardless of what any downstream agent has done) instead of any agent's history. Verified against the real run's actual saved design file: correctly reproduces `{'FE': 38, 'BE': 38, 'OPS': 22}`. `server/pipeline_runner.py` doesn't need the equivalent, since the web UI never resumes internally (resuming a web-started run goes through the CLI's `--resume`). If you touch turn-budget resume logic again: a mock-based test alone won't catch this class of bug (see `tests/test_turn_budget.py`'s `ReapplyTurnBudgetOnResumeTests` for what it *does* cover) — the failure mode only appears under a real concurrent crash, so re-verify live.
- **Skipping an engineer with no real work, instead of always running all three** — a real, observed failure mode: `ARCHITECT_PROMPT` used to open with "a production-grade technical design that **three engineers** ... can implement", which had no notion of a role needing zero work. Live testing against a literal one-file static "Hi" page confirmed the architect reliably invented busywork to satisfy that framing — a backend HTML-validator/contract-test Node toolchain and a full Terraform/CI/Docker deployment for a page that's one HTML file — each of which is a full paid Claude Code session. QA correctly caught real defects in the manufactured BE toolchain (wrong relative paths, a broken test-discovery glob), triggering a rework loop that then hit the account-wide Claude Code session limit before finishing — i.e. the over-engineering wasn't just wasteful, it was what caused that run to fail. Root cause was the *prompt*, not the orchestration: `pipeline.py`'s graph unconditionally fans out to FE/BE/OPS on every run (no "skip" edge condition exists, or could easily exist — GraphFlow's activation-count model is computed once from static graph topology at `build_team()` time, before the architect has run, so it can't conditionally omit a node whose necessity isn't known until then). Fixed at the prompt + agent level instead, not by changing the graph: `ARCHITECT_PROMPT`/`SPEC.md` (kept in sync as always) now explicitly permit and expect zero tagged tasks for a role that doesn't need one ("up to three" engineers, not always three), and require a turn budget of exactly `0` in that case — reusing the *existing* Turn Budget Estimate line as the skip signal rather than inventing new parsing. `parse_turn_budget()` (`pipeline.py`) special-cases `0` to bypass the `[MIN_ENGINEER_TURNS, MAX_ENGINEER_TURNS]` clamp (which would otherwise silently turn "no work" back into a real 8-turn session). `ClaudeCodeAgent.on_messages()` (`claude_code_agent.py`) checks `self._allowed_tools and self._max_turns == 0` right after accepting the turn and, if true, returns a canned "no tasks assigned" response immediately — without constructing `ClaudeAgentOptions` or calling `claude_agent_sdk.query()` at all, so a skipped role costs nothing. The `self._allowed_tools` guard is required: PM/Architect/UAT/Reporter also get `max_turns` values (as a runaway guard, not a work budget — see the turn-budget bullet above) and must never be treated as skippable. GraphFlow's own routing is untouched by this: the skipped role still returns exactly one `Response` message, satisfying the same edge/activation bookkeeping (e.g. QA's fan-in) a real session would — the optimization is purely "don't spend a real session," not "don't participate in the graph." **Not exercised by `tests/mock_agent.py`'s `ScriptedClaudeCodeAgent`** (it overrides `on_messages()` wholesale, so the skip check in the real class never runs through it — same reasoning as the `error_max_turns` note above); covered instead by `tests/test_skip_idle_engineer.py`, which calls the real `ClaudeCodeAgent.on_messages()` directly with `claude_agent_sdk.query` patched to raise if ever called, so a regression fails loudly and instantly rather than silently spending quota.
- **Max-turns auto-bump + truncation banner** (`claude_code_agent.py`): a real logged run showed all three engineers hitting `max_turns` on the first pass, then `backend_engineer` hitting the *same* ceiling again on its QA-rework turn — the architect's budget is applied once and nothing revised it. `_last_hit_max_turns` persists across turns (unlike the local `hit_max_turns`); the next `on_messages()` call consumes it and bumps `_max_turns` by +50% (floor +8), clamped to `_MAX_TURN_BUDGET = 50` (mirrors `pipeline.MAX_ENGINEER_TURNS`; duplicated to avoid a circular import — keep in sync). One-shot: cleared on consumption, so clean turns don't inflate. Like `set_max_turns()`, it's in-memory only — not in `save_state()`, resets on `--resume`. Separately, a truncated turn's `final_text` gets a prepended WARNING banner: the transcript of a cut-off session is progress *narration* that can claim unfinished work as done (observed live: "Now tests: ... covering happy path" for test files never written), and it becomes the persisted `.pipeline-docs/` artifact QA reads. Banner is prepended *before* `_apply_verdict_safety_net()` so QA/UAT last-line verdicts still route. Covered by `tests/test_max_turns_budget_bump.py` (patched `query`, no quota).
- **Artifact schemas / message protocol** (`src/looper/artifacts.py`): every role's OUTPUT ARTIFACT now has a machine-checkable `ArtifactSpec` (kind, H1 title pattern, required/optional H2 sections, verdict tokens). `validate_artifact()` is **advisory** — warnings printed by `main.py`, recorded as `artifact_warning` events by `server/pipeline_runner.py` — never fatal (an LLM consumer usually copes with a missing heading; killing a paid half-done run over one is worse). Keep `ARTIFACT_SPECS` in sync with the OUTPUT ARTIFACT blocks in `prompts.py`/SPEC.md, same rule as `context_sources`. Section matching is prefix + case-insensitive on purpose; the zero-budget skip message (different H1, no OPS Runbook) validates clean via its `_SKIP_MARKER`. `find_blockers()` extracts GLOBAL_RULES' `BLOCKER: <reason>` escalation lines (surfaced as `blocker_raised` events / printed) — nothing routes on them. Validated against a real run's artifacts: correctly flags only the max-turns-truncated engineer message.
- **Cross-run memory** (`src/looper/run_memory.py`): `<output>/memory/runs.jsonl`, one record per completed run (keyed by stamp; re-recording a stamp replaces, so a resumed run doesn't double-count), aggregated by `summarize_run()` from the run's existing `sdk-interactions.jsonl` — no new instrumentation. `calibration_hint()` turns history into an architect prompt addendum ("N of M engineer sessions ran out of budget") passed via `build_team(architect_addendum=...)` — runtime-composed context like the TURN BUDGET line, deliberately NOT an edit to `ARCHITECT_PROMPT`/SPEC.md. Recorded only on the completion path (both entry points share `output/memory/`); permanently-failed runs aren't recorded. Tests: `tests/test_run_memory.py`.
- **Bounded session concurrency** (`session_limiter` on `ClaudeCodeAgent`; env `LOOPER_MAX_PARALLEL_SESSIONS`, unset/0 = unbounded): one `asyncio.Semaphore` created in `build_team()` and shared by every agent caps concurrent real SDK sessions — only the FE/BE/OPS fan-out ever exceeds one. Exists because three concurrent engineer sessions can trip the account-wide session limit mid-fan-out (the exact crash `recover_stuck_agents()` recovers from — the limiter avoids needing that recovery, trading wall-clock time). Acquired around only the `query()` iteration (skip path and prompt building stay outside), released in a `finally` so a crashed session can't deadlock the team. GraphFlow dispatch is untouched; waiting agents queue inside their turn. Tests: `tests/test_session_limiter.py` (proves real overlap without the limiter, none with it, and release-on-crash).
- **Scope validator** (`scope_validator`, SPEC.md 3.1b): annotate-only gate between PM and architect — always forwards (both edges unconditional; no reject loop, no new activation groups, deliberately). Enforcement is distributed across downstream prompts, which is why touching any of them requires knowing about the others: the architect creates NO tasks for the SCOPE REVIEW's Must-Cut items, **QA marks them TRIMMED instead of FAIL** (without this, a trimmed AC-n would trip QA's "no implementing code = automatic failure" rule and loop forever), and UAT treats a *wrong* trim as re-scope grounds rather than an engineer defect. Exists because PM-stage scope inflation was a real, measured incident (a "beautiful car showroom website" goal → $13.39 full-stack platform, whose padded scope also ran the engineers out of turn budgets). Runs again on a UAT re-scope loop (the re-groomed PRD passes back through the gate); does NOT run on QA rework loops. Its SCOPE REVIEW routes to architect/QA/UAT `context_sources` only — engineers get scope via the TECH DESIGN. Tests: `tests/test_scope_validator.py`.
- **KPI report** (`src/looper/report.py`, `python -m looper.report`): aggregates `output/memory/runs.jsonl` + each run's `sdk-interactions.jsonl` into per-run/per-role costs, turn-budget calibration (uses the `num_turns` field `_log_interaction()` now records — older log entries lack it and are skipped for the ratio), and defect stats via `artifacts.parse_defects()` (tolerant markdown extraction of D-n/severity/owner — the pragmatic alternative to forcing JSON artifacts; dedupes by id per run since ids are immutable across rework loops). Read-only. Tests: `tests/test_report.py`; the parser is verified against a real run's QA reports.
- **Visual QA** (architect-gated Playwright verification, `ARCHITECT_PROMPT`'s "Visual QA" section / `QA_PROMPT` step 4): text-only code review cannot catch layout/rendering defects or confirm an animation-heavy frontend actually renders — but a real render-and-screenshot pass costs real turns, so it's opt-in per goal rather than always-on (mirrors the FE/BE/OPS zero-budget skip's "don't pay for work a goal doesn't need" philosophy). The architect states exactly one line, `VISUAL_QA: YES: <N> — <reason>` or `VISUAL_QA: NO — <reason>` (N = extra turns QA needs on top of its baseline `QA_MAX_TURNS`); `parse_visual_qa_extra_turns()`/`apply_visual_qa_budget()` (`pipeline.py`) parse and bump `qa_engineer`'s budget via the same `set_max_turns()` seam as FE/BE/OPS, clamped to `MAX_VISUAL_QA_EXTRA_TURNS = 30`. Deliberately a SEPARATE function from `apply_turn_budget_from_architect()` (different section of the same TECH DESIGN message, governing a different role) but always called alongside it at every call site — `main.py`'s live `on_message`, `server/pipeline_runner.py`'s, and `reapply_turn_budget_on_resume()` (which now also re-derives QA's visual budget from the on-disk design on `--resume`, for the identical in-memory-state-doesn't-survive-a-crash reason FE/BE/OPS's budget needed it). No new `allowed_tools` grant needed — QA already has `Bash` (build/serve/Playwright) and `Read` (which already handles image files, so QA `Read`s its own screenshots to visually inspect them). Added `ClaudeCodeAgent.max_turns` as a read-only property (the getter half of `set_max_turns()`) specifically so `apply_visual_qa_budget()`'s "current baseline + extra" callers don't reach into `_max_turns` directly, which `set_max_turns()`'s own docstring already said not to do. **Honest limit, stated explicitly in QA_PROMPT itself so QA doesn't overclaim it**: a screenshot is a frozen frame and a short recording is a coarse signal — this catches real layout/rendering defects and confirms an interaction visibly fires, it does NOT and CANNOT verify subjective animation "feel," smoothness, or premium polish. Not yet verified live (the actual Playwright build-serve-screenshot-Read workflow needs a real run to confirm QA's Bash sandbox can install/drive a headless browser end to end) — treat like the `error_max_turns` SDK-transport fix: re-verify live before trusting it, a mock can't exercise a real browser. Tests: `tests/test_visual_qa.py` (parsing/budget-application logic only).
- **Persistent Playwright browser cache** (`ClaudeCodeAgent.extra_env` / `build_team(output_dir=...)`): re-downloading Chromium every run (or, worse, every Dockerized container restart) is wasteful — `qa_engineer` gets `PLAYWRIGHT_BROWSERS_PATH` pointed at `<output_dir>/.playwright-browsers/`, a location OUTSIDE the per-run `workspace/<stamp>/` dir so it survives across every run, not just one. Confirmed by reading `claude_agent_sdk`'s own `subprocess_cli.py` `connect()`: the SDK subprocess already inherits the full host environment by default (`{**os.environ, ...options.env}`, `options.env` only *adding to* it) — so this already worked by accident on a long-lived host via the OS's own default Playwright cache dir, but that's fragile exactly where it matters most: `deploy/`'s Dockerized web UI has no such implicit cache on a fresh container, UNLESS it happens to live inside the already-mounted `output/` volume, which `<output_dir>/.playwright-browsers/` now guarantees for free — no `deploy/docker-compose.yml` changes needed. `ClaudeCodeAgent.__init__` gained a generic `extra_env: Mapping[str, str] | None` param (merged into `ClaudeAgentOptions(env=...)`) rather than a Playwright-specific one, in case another role needs a similar per-agent env override later; only QA gets one set today. `build_team()` gained a matching `output_dir: Path | None` param (both `main.py` call sites and `server/pipeline_runner.py`'s pass it; `None` — the default for every existing test call site — means no override, backward compatible). `output/` is already fully gitignored, so the cache directory needs no new `.gitignore` entry. Tests: `tests/test_visual_qa.py`'s `PersistentBrowserCacheTests` (path correctness, QA-only scoping, backward-compat default).
- **Dollar-budget caps** (`LOOPER_MAX_SESSION_BUDGET_USD` / `LOOPER_MAX_RUN_BUDGET_USD`, both unset = uncapped): per-SESSION cap goes to `ClaudeAgentOptions.max_budget_usd` — the SDK stops an exceeded session with an `error_max_budget_usd` result, which `ClaudeCodeAgent` handles exactly like `error_max_turns` (partial output + truncation banner, never a crash), with one deliberate difference: it does NOT arm the `_last_hit_max_turns` bump, since more turns can't fix a dollar constraint. The presumed second CLI exception after it (symmetric with max_turns' shell-consumers exit) is gated the same way but NOT live-verified for the budget subtype — noted inline in `on_messages()`. Run-level cap lives in `runner.FailFastMonitor` (the one place that sees every `turn_completed` event's `cost_usd`): cumulative spend crossing the cap trips the same kill switch a crashed agent uses — checkpointed, resumable, seedable on resume (`seed_spent()`, derived from the SDK log so the cap covers the whole run across resumes). Checked after a completed turn, not before: the in-flight turn's money is already spent; the cap prevents the NEXT session. Tests: `tests/test_budget_caps.py`.
- **Cross-run defect memory** (`run_memory.defect_history_hints()` / `build_team(role_addenda=...)`): BLOCKER/MAJOR defects from past runs' QA reports (parsed via `artifacts.parse_defects()`, which now also captures a cleaned `summary` snippet per defect) become per-engineer prompt addenda — "your role produced these defects in earlier runs, don't repeat the class of mistake". MINOR defects excluded by design (hints are for rework-loop-costing mistakes; nitpicks dilute the signal). Same runtime-composed-addendum pattern as the architect calibration hint; `role_addenda` is generic (any role name → appended text) with QA's defect hints its only current user. Both entry points wire it. Tests: `tests/test_run_memory.py`'s `DefectHistoryHintsTests`.
- **Deploy verification** (architect-gated, ARCHITECT_PROMPT step 9 / QA_PROMPT step 5 / a new OPS self-build rule): the third gate alongside Visual QA — `DEPLOY_VERIFY: YES: <N>`/`NO` grants QA extra turns to `docker compose build`, `up -d`, hit the design's health endpoints + one real request path against RUNNING containers, and tear down (`down -v`). OPS additionally must build its own images before finishing when the daemon is available ("a Dockerfile that has never been built is not a deliverable" — a real run shipped a backend whose image couldn't build, caught only at QA). Docker-daemon absence is explicitly SKIPPED-environment, never a defect — a `YES` is always safe to grant. Budget machinery mirrors Visual QA exactly (`parse_deploy_verify_extra_turns()`/`apply_deploy_verify_budget()`, `MAX_DEPLOY_VERIFY_EXTRA_TURNS = 25`, stacks cumulatively with the visual extra — called after it with QA's then-current budget as base — and survives resume under the `"QA_DEPLOY"` key). NOT live-verified yet, same status Visual QA had before its live test. Tests: `tests/test_visual_qa.py`'s `DeployVerifyBudgetTests`.
- **Graceful degradation at rework exhaustion** (`QaVerdictRouter` in `pipeline.py`, UAT_PROMPT step 3): the 3rd QA_FAIL no longer hard-kills the run with PIPELINE_FAILED — a stateful router on QA's outgoing edges sends fails #1-#2 to the engineers and the final one to UAT, whose prompt makes the ship-with-Known-Issues / reject call (open BLOCKER = never approvable; reject = the normal one re-scope loop). **The old `TokenCountTermination("QA_FAIL", max_count=MAX_QA_LOOPS)` was RAISED to `+2` as a pure runaway guard — putting it back at MAX_QA_LOOPS would kill the run at the exact moment the router is routing to UAT.** Router mechanics worth knowing before touching: both edge conditions can fire for the same QA message, so the fail counter registers each message object once (`id()`-keyed; safe because GraphFlow retains messages for the run's lifetime). Two documented limits, both bounded by the safety terminations: the count is cumulative across UAT re-scopes (a re-scoped attempt that fails QA goes straight to judgment), and it's in-memory only (a `--resume` resets it, allowing up to 2 extra rework loops). Tests: `tests/test_pipeline.py`'s exhausted-rework tests.
- **Autonomous crash recovery** (`src/looper/recovery.py` + main.py's auto-resume loop; `LOOPER_AUTO_RESUME`, default ON, `LOOPER_MAX_AUTO_RESUMES` default 3): built from the real session-limit incident — the error text ("You've hit your session limit · resets 3:30am (Asia/Calcutta)", live-observed format) contains everything needed to wait and resume automatically. `compute_retry_delay()`: session limits wait until the parsed reset (+120s buffer, capped at 6h, 30min fallback if the time doesn't parse); anything else gets a short bounded backoff (60s, 300s, then give up). The loop reuses the SAME `_restore_from_checkpoint()` helper as manual `--resume` (extracted so the two paths can't drift — every resume repair bug from this file's history applies to both) and re-arms the monitor via `FailFastMonitor.reset()` (clears the kill switch, keeps cumulative spend). Also adds the per-run PID lock (`output/locks/<stamp>.lock`) closing the documented double-resume hazard: a second invocation against a run whose holder is still alive refuses to start; a stale lock (dead PID) is taken over automatically. `LOOPER_AUTO_RESUME=0` restores the old print-instructions-and-exit behavior. The end-to-end loop is covered by a scripted integration test that drives `main.run()` itself through crash → auto-restore → completion (`tests/test_recovery.py`); the session-limit wait math is unit-tested against the literal observed error string. Not yet exercised against a real session-limit event — the next real one is the live test.

Load-bearing since cross-run memory landed: a scripted test run recording into the real `output/memory/runs.jsonl` would leak test data into the architect calibration hint used by real runs (happened once; cleaned).

An optional FastAPI + React app that wraps the same `build_team()` behind a multi-user web control plane — see README's [Web UI](README.md#web-ui) section for the full writeup (architecture, how to run it, Phase-1 limitations named explicitly). Load-bearing details for anyone touching this code:

- `server/pipeline_runner.py` builds the team with `on_event` wired to `_record_event()`, which writes each event to SQLite (`RunEvent`) and publishes it to `server/events.py`'s in-process broker — both in that order, and both must stay in sync with each other (an event visible in one but not the other is a bug).
- **`GET /runs/{id}/events` (`server/routes/runs.py`) must subscribe to the broker *before* querying event history, not after.** Querying history first and subscribing second leaves a real gap: any event published between those two steps is dropped forever — absent from the snapshot already taken, missed by a subscription that doesn't exist yet. This was a live, reproduced bug (scripted `product_manager` completed its turn faster than the network round-trip to open the SSE connection, and its events vanished entirely), not a hypothetical one — fixed by subscribing first and deduping the merged stream by each event's monotonic `seq`. `tests/test_server_e2e.py` asserts `seq` comes back gap-free and duplicate-free specifically to catch a regression here.
- SSE auth is intentionally different from every other route: `/runs/{id}/events` reads the JWT from a `?token=` query param (`get_current_user_sse` in `server/deps.py`) instead of an `Authorization` header, because a browser's native `EventSource` cannot set custom headers. Don't "fix" this to match the other routes — it would break the frontend's `EventSource` connection.
- `asyncio.create_task()` is called directly inside the async `POST /runs` route handler, not via FastAPI's `BackgroundTasks` — `BackgroundTasks` runs sync callables through `run_in_threadpool`, a worker thread with no running event loop, where `asyncio.create_task()` raises `RuntimeError: no running event loop`. The task is kept in a module-level `_background_tasks: set[asyncio.Task]` with a `done_callback` purely so it isn't garbage-collected mid-run (asyncio only holds a weak reference otherwise) — not for any other bookkeeping purpose.
- `server/auth.py` calls `bcrypt.hashpw`/`bcrypt.checkpw` directly, not through `passlib`. `passlib==1.7.4` is unmaintained and incompatible with `bcrypt>=4.1`'s stricter 72-byte handling — it raised `ValueError` on password hashing even for short passwords, via a broken internal self-test during its bcrypt-backend detection. Don't reintroduce `passlib` here.
- **Deployment** (`deploy/`): Dockerfiles for API + web, compose, nginx SPA config — see `deploy/README.md` for the operational constraints (claude CLI OAuth must be bind-mounted from the host, single API replica only while the SSE broker is in-process). CORS origins are env-driven (`LOOPER_CORS_ORIGINS` in `server/app.py`); the frontend's API base is a Vite **build-time** arg (`VITE_API_BASE`), must be browser-reachable, not container-to-container. Image builds not yet verified against a live Docker daemon (compose config syntax is).

## Testing

- `tests/` tests *orchestration* (routing, rework loops, hard-stop termination, checkpoint/resume) against `tests/mock_agent.py`'s `ScriptedClaudeCodeAgent` — a `ClaudeCodeAgent` subclass that overrides only `on_messages()` to return scripted text instead of calling `claude_agent_sdk`. `build_team()` takes `agent_cls` for exactly this seam; production code never passes it. See README's [Testing](README.md#testing) section for the full writeup, including a documented test-design bug worth reading before writing a similar test: simulating "an interrupted run" with `break` on the message stream does **not** work — `GraphFlow`'s runtime is a producer/consumer queue that keeps processing in the background regardless of whether the consumer is still reading, and with near-instant scripted responses it races the whole pipeline to completion before `save_state()` gets called. Use the `CRASH` script sentinel (an actual raised exception) to simulate an interruption instead — that's what genuinely halts the runtime, matching how a real Claude Code session-limit failure behaves in production.
- `tests/test_server_e2e.py` covers the web UI backend the same way — scripted agent, no real Claude Code sessions — using an `httpx.ASGITransport` client against the real FastAPI `app` and a throwaway file-backed SQLite DB (`tempfile`, not `sqlite:///:memory:` — a bare in-memory URL gives each pooled connection its own separate database under SQLAlchemy's default pooling, so tables created in `setUp` become invisible to the next connection).
- `tests/test_max_turns_handling.py` unit-tests `_apply_verdict_safety_net()` directly (no SDK, no mock agent needed — it's pure text logic). It does **not** cover the `error_max_turns` double-exception behavior described above — that's SDK-transport-level and was verified with a real, deliberately-undersized `max_turns` live session instead, not a mock. If you touch that code path again, re-verify live; a mock can't authentically reproduce it since `ScriptedClaudeCodeAgent` overrides `on_messages()` wholesale and never calls `claude_agent_sdk.query()`.
- `tests/test_turn_budget.py` covers `parse_turn_budget()`/`apply_turn_budget_from_architect()`/`find_message_from()` — all pure Python logic, no SDK needed, so this is the fast-suite coverage for the dynamic turn budget. What it does **not** cover, because a mock can't: that the architect actually produces a parseable "Turn Budget Estimate" section in practice, and that `set_max_turns()` really does change what the real SDK session's system prompt says — both were confirmed with real, live Claude Code sessions instead (see README "Evidence this works"). Re-verify live if you change `ARCHITECT_PROMPT`'s budget-estimate wording or the system-prompt injection in `on_messages()`.

## Conventions

- Verify AutoGen API signatures against the installed version before changing orchestration code — the GraphFlow API is experimental and shifts between minor releases.
- No secrets in code; provider/model selection only via env vars (see `.env.example`).
- `ClaudeCodeAgent` gives FE/BE/OPS/QA real Bash access with `acceptEdits` (auto-approved, no human in the loop). The `PreToolUse` guard (above) blocks a specific denylist and out-of-workspace writes, but it is a string/path check enforced in-process, not an OS sandbox (`claude_agent_sdk.SandboxSettings` would be that, and isn't wired in) — treat the workspace as untrusted-generated-code territory, not as safe as the rest of the repo.
