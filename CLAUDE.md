# AiTeam

Autonomous AI software delivery team on AutoGen AgentChat (0.7.x) `GraphFlow`. Eight agents (PM, Architect, FE/BE/OPS engineers, QA, UAT, Release Reporter) run a deterministic pipeline with QA rework loops (max 3) and a single UAT re-scope loop. The authoritative behavior spec is `SPEC.md` — keep prompts and routing consistent with it. The Release Reporter exists partly for a structural reason: GraphFlow validation requires at least one leaf node, and the run completes naturally when it executes.

## Commands

```sh
.venv/bin/pip install -r requirements.txt          # deps
PYTHONPATH=src .venv/bin/python -m aiteam.main "<goal>"   # run pipeline
PYTHONPATH=src .venv/bin/python -m aiteam.main --resume output/checkpoints/<stamp>.json   # resume a failed/interrupted run
PYTHONPATH=src:. .venv/bin/python -m unittest discover -s tests -t . -v   # orchestration tests (no Claude Code sessions, instant)

PYTHONPATH=src:. .venv/bin/uvicorn server.app:app --reload --port 8000   # web UI backend
cd web && npm install && npm run dev                                     # web UI frontend (http://localhost:5173)
```

All eight agents run via `claude-agent-sdk`, which shells out to the `claude` CLI (Claude Code) and uses **its own OAuth login** — no `ANTHROPIC_API_KEY`/`OPENAI_API_KEY` needed. Requires the `claude` CLI installed and authenticated on the host (`claude auth status`). Optional `AITEAM_CODE_MODEL` overrides the model all agents use (default: Claude Code's own default).

`src/aiteam/config.py` (`make_model_client`, `AITEAM_PROVIDER`) is unused dead code, kept intentionally in case the pipeline is ever wired back to a raw-API-key provider (`AnthropicChatCompletionClient`/`OpenAIChatCompletionClient` via `autogen-ext`) instead of the `claude` CLI. It is not called from `pipeline.py` or `main.py` — don't assume it's live.

## Architecture

- `src/aiteam/pipeline.py` — the graph. Key invariant: nodes with multiple incoming edges (engineers: architect + QA-rework; PM: entry + UAT-reject) must keep both edges in a shared `activation_group` with `activation_condition="any"`, or the first pass deadlocks (GraphFlow defaults to "all").
- Routing is by verdict tokens (`QA_PASS`/`QA_FAIL`, `UAT_APPROVED`/`UAT_REJECTED`) matched against the **last non-empty line** of the agent's message (`verdict_is` / `last_line`) — never substring-match, since prompts/templates can mention both tokens.
- `src/aiteam/termination.py` — `TokenCountTermination` hard-stops after 3 `QA_FAIL`s with a `PIPELINE_FAILED` stop reason.
- `src/aiteam/prompts.py` — verbatim from SPEC.md section 3 (including the "token must appear nowhere else" hardening on QA/UAT). Edit SPEC.md and prompts.py together; they must not drift.
- `src/aiteam/claude_code_agent.py` — `ClaudeCodeAgent`, a `BaseChatAgent` that runs its role as a real `claude_agent_sdk` session (`permission_mode="acceptEdits"`) instead of a plain model-client completion. Every agent is one of these. PM, Architect, UAT, and Release Reporter get `allowed_tools=[]` (pure text reasoning, no filesystem access); FE and BE get file/bash tools scoped to `workspace/frontend/`/`workspace/backend/`; OPS gets the workspace root; QA gets `Read`/`Glob`/`Grep`/`Bash` only (**no** `Write`/`Edit` — it verifies, it doesn't modify the code it's independently checking) scoped to the root. See README's "Access per role" table for the full breakdown. Each `query()` call is a fresh Claude Code session with no memory of prior turns, so `ClaudeCodeAgent` replays the accumulated upstream chat history as the prompt every turn (mirroring `AssistantAgent`'s internal `model_context`) — but code from earlier turns persists for the tool-using agents because it's really on disk in the workspace.
- **`tools=` is what restricts availability — `allowed_tools=` does not, despite the name.** Confirmed live: a session built with `allowed_tools=["Read"]` and no `tools` restriction could still run `Bash`. Per `ClaudeAgentOptions`'s own docstrings, `allowed_tools` only skips the permission prompt for tools already available; `tools` (or `disallowed_tools`) is what actually removes a tool from the model's context. `ClaudeCodeAgent.on_messages()` sets both `tools=self._allowed_tools` and `allowed_tools=self._allowed_tools` for this reason — before this fix, every role (including the four "tool-less" reasoning agents) had Claude Code's full default toolset available the whole time, gated only by whether the prompt happened to ask for it. If you add a new tool restriction anywhere, set `tools=`, not just `allowed_tools=`.
- **QA's "no Write/Edit" is not airtight while it still has Bash.** Verified live: with `Write`/`Edit` removed from `tools`, a QA-like agent correctly refused a `Write` call — but then offered to create the same file via `Bash` instead (e.g. `echo ... > file`), since Bash can replicate file-writing and isn't blocked by the `_BASH_DENY_PATTERNS` denylist (blocking generic redirection would also block QA's legitimate build/test output). QA needs Bash to run real tests, so this residual gap is inherent, not a bug to fix trivially — treat it as a prompting-level expectation (QA's prompt says its job is to verify, not modify), not a hard guarantee.
- **Permission guard** (`_make_hooks` in `claude_code_agent.py`): a `PreToolUse` hook, not `allowed_tools` or `can_use_tool`, enforces per-call restrictions (Write/Edit path containment to the agent's own `cwd`; a Bash command denylist). Load-bearing detail if you touch this: `can_use_tool` was tried first and doesn't work — an `allowed_tools` entry that grants a whole tool (e.g. plain `"Bash"`) auto-approves every call before `can_use_tool` is ever consulted (`CanUseToolShadowedWarning`), confirmed live when an unrestricted `curl` call sailed straight through it. `PreToolUse` hooks aren't shadowed this way, which is the entire reason the guard is built on hooks instead. If you add a new restriction, add it as a hook, not a `can_use_tool` callback.
- **Context routing** (`context_sources` on `ClaudeCodeAgent`, configured per role in `pipeline.py`): GraphFlow broadcasts every message to every agent, so an unfiltered history replay grows with the whole transcript and re-sends artifacts a role's prompt says it doesn't consume (plus superseded PRDs/QA reports on loops — a token *and* stale-context problem). Each role's replayed prompt is filtered to the sources its SPEC.md "Input:" line names, keeping only the latest message per source. Filtering happens at prompt-build time (`_build_prompt()`) — `_history` and checkpoints stay complete. Keep the `context_sources` dicts in sync with each role's "Input:" line if either changes.
- **Pointer routing** (`pointer_files` on `ClaudeCodeAgent`; tool-using roles only): `main.py` persists each agent's latest message to `<workspace>/.pipeline-docs/<source>.md` (dir name shared via `pipeline.ARTIFACT_DIR_NAME` — keep `main.py` and `pipeline.py` agreeing on it). On a turn where a pointer-listed artifact did **not** arrive in that turn's delta and its file exists, `_build_prompt()` replays a one-line pointer to the file instead of the full text; the agent Reads/Greps back only what it needs. The effectiveness guardrail is load-bearing: an artifact is always inlined in full the first time it reaches an agent (first pass, or a redesign after UAT re-scope — it's in the delta both times), so pointers can never hide a contract the agent hasn't seen; missing file falls back to inline; `_build_prompt()` with no delta info never pointers. Never set `pointer_files` on a tool-less role — no Read tool means a pointer is a dead end. Measured on a second-QA-round engineer turn: unfiltered replay 26.3K chars → context routing 11.1K (−58%) → routing + design pointer 3.3K (−87%); realized savings are somewhat less since the agent reads back the sections it needs, which is the point — spend scales with what the defects touch, not with design size.
- Workspace layout, built per run in `main.py` under `<output_dir>/workspace/<timestamp>/` and passed into `build_team()`: `frontend/` (FE's cwd), `backend/` (BE's cwd), and the root (OPS's and QA's cwd, so OPS can reference `./frontend` and `./backend` and QA can read/test everything; also the cwd for the tool-less reasoning agents, who never use it). This convention is encoded into the FE/BE/OPS/QA prompts in `prompts.py`/`SPEC.md` — the path names and pipeline.py's directory assignment must stay in sync.
- **Checkpoint/resume** (`main.py`): after every completed agent turn, `main.py` calls `GraphFlow.save_state()` and overwrites `<output_dir>/checkpoints/<stamp>.json` (goal, workspace path, full team state). On any exception (Claude Code session limit hit, crash, etc.) the run stops with that path printed — rerun with `--resume <path>` to continue from the last completed turn instead of restarting at PM. Verified: a freshly-constructed team (new `GraphFlow`, new agent instances) correctly resumes at the next graph node after `load_state()`, without re-running completed ones. This relies on two things staying correct together: `GraphFlow.save_state()`/`load_state()` (built into AutoGen, captures graph position + each participant's pending message buffer) and `ClaudeCodeAgent.save_state()`/`load_state()` (added here — `BaseChatAgent`'s default is a stateless no-op, so without this override the agent's replayed `_history` would not survive a resume even though the graph position would). Resuming calls `run_stream(task=None)` — per AutoGen's own contract, this continues the previous task rather than starting a new one.
- **`on_event` hook** (`OnEvent` type alias in `claude_code_agent.py`): an optional constructor param on `ClaudeCodeAgent`, fired via `self._emit(event_type, detail, **extra)` at `turn_started`, each `tool_call`, `turn_completed`, and `error`. `_emit` swallows exceptions from the callback itself — a live-status sink must never break an actual pipeline turn. `build_team()` threads `on_event` through to every `agent_cls(...)` call; `main.py` never passes one (`None`, a no-op), so the CLI path is unaffected. This exists for `server/pipeline_runner.py` (below) to record live per-agent progress; `tests/mock_agent.py`'s `ScriptedClaudeCodeAgent` also calls `_emit` at the same points, so the event stream is unit-testable without spending Claude Code quota.

## Web UI (`server/` + `web/`)

An optional FastAPI + React app that wraps the same `build_team()` behind a multi-user web control plane — see README's [Web UI](README.md#web-ui) section for the full writeup (architecture, how to run it, Phase-1 limitations named explicitly). Load-bearing details for anyone touching this code:

- `server/pipeline_runner.py` builds the team with `on_event` wired to `_record_event()`, which writes each event to SQLite (`RunEvent`) and publishes it to `server/events.py`'s in-process broker — both in that order, and both must stay in sync with each other (an event visible in one but not the other is a bug).
- **`GET /runs/{id}/events` (`server/routes/runs.py`) must subscribe to the broker *before* querying event history, not after.** Querying history first and subscribing second leaves a real gap: any event published between those two steps is dropped forever — absent from the snapshot already taken, missed by a subscription that doesn't exist yet. This was a live, reproduced bug (scripted `product_manager` completed its turn faster than the network round-trip to open the SSE connection, and its events vanished entirely), not a hypothetical one — fixed by subscribing first and deduping the merged stream by each event's monotonic `seq`. `tests/test_server_e2e.py` asserts `seq` comes back gap-free and duplicate-free specifically to catch a regression here.
- SSE auth is intentionally different from every other route: `/runs/{id}/events` reads the JWT from a `?token=` query param (`get_current_user_sse` in `server/deps.py`) instead of an `Authorization` header, because a browser's native `EventSource` cannot set custom headers. Don't "fix" this to match the other routes — it would break the frontend's `EventSource` connection.
- `asyncio.create_task()` is called directly inside the async `POST /runs` route handler, not via FastAPI's `BackgroundTasks` — `BackgroundTasks` runs sync callables through `run_in_threadpool`, a worker thread with no running event loop, where `asyncio.create_task()` raises `RuntimeError: no running event loop`. The task is kept in a module-level `_background_tasks: set[asyncio.Task]` with a `done_callback` purely so it isn't garbage-collected mid-run (asyncio only holds a weak reference otherwise) — not for any other bookkeeping purpose.
- `server/auth.py` calls `bcrypt.hashpw`/`bcrypt.checkpw` directly, not through `passlib`. `passlib==1.7.4` is unmaintained and incompatible with `bcrypt>=4.1`'s stricter 72-byte handling — it raised `ValueError` on password hashing even for short passwords, via a broken internal self-test during its bcrypt-backend detection. Don't reintroduce `passlib` here.

## Testing

- `tests/` tests *orchestration* (routing, rework loops, hard-stop termination, checkpoint/resume) against `tests/mock_agent.py`'s `ScriptedClaudeCodeAgent` — a `ClaudeCodeAgent` subclass that overrides only `on_messages()` to return scripted text instead of calling `claude_agent_sdk`. `build_team()` takes `agent_cls` for exactly this seam; production code never passes it. See README's [Testing](README.md#testing) section for the full writeup, including a documented test-design bug worth reading before writing a similar test: simulating "an interrupted run" with `break` on the message stream does **not** work — `GraphFlow`'s runtime is a producer/consumer queue that keeps processing in the background regardless of whether the consumer is still reading, and with near-instant scripted responses it races the whole pipeline to completion before `save_state()` gets called. Use the `CRASH` script sentinel (an actual raised exception) to simulate an interruption instead — that's what genuinely halts the runtime, matching how a real Claude Code session-limit failure behaves in production.
- `tests/test_server_e2e.py` covers the web UI backend the same way — scripted agent, no real Claude Code sessions — using an `httpx.ASGITransport` client against the real FastAPI `app` and a throwaway file-backed SQLite DB (`tempfile`, not `sqlite:///:memory:` — a bare in-memory URL gives each pooled connection its own separate database under SQLAlchemy's default pooling, so tables created in `setUp` become invisible to the next connection).

## Conventions

- Verify AutoGen API signatures against the installed version before changing orchestration code — the GraphFlow API is experimental and shifts between minor releases.
- No secrets in code; provider/model selection only via env vars (see `.env.example`).
- `ClaudeCodeAgent` gives FE/BE/OPS/QA real Bash access with `acceptEdits` (auto-approved, no human in the loop). The `PreToolUse` guard (above) blocks a specific denylist and out-of-workspace writes, but it is a string/path check enforced in-process, not an OS sandbox (`claude_agent_sdk.SandboxSettings` would be that, and isn't wired in) — treat the workspace as untrusted-generated-code territory, not as safe as the rest of the repo.
